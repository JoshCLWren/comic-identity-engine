## Problem Analysis

### What's Parallel?
вң… Cross-platform search within a row (6 platforms simultaneously via `PlatformSearcher`)
вң… Multiple workers in pool

### What's Serial?
вқҢ **Row processing** - only ONE worker processes the entire CSV
вқҢ **Task granularity** - ONE giant task for all 5,200 rows

### Why Workers Sit Idle
- Worker 1 picks up the single `import_clz_task`
- Workers 2-10 have nothing to do
- Worker 1 processes 5,200 rows serially
- Workers 2-10 wait hours for Worker 1 to finish

## Current Implementation Issues

### Issue 1: Task Granularity Too Large
```python
# ONE task for entire CSV
await queue.enqueue_import_clz(csv_path, operation_id)

# Should be:
for chunk in chunks:
    await queue.enqueue_import_clz_chunk(chunk, operation_id)
```

### Issue 2: Serial Row Processing
```python
# Current (WRONG):
for idx, row in enumerate(rows):  # Serial!
    result = await resolver.resolve_issue(...)

# Should be:
tasks = [process_row(row) for row in rows]
results = await asyncio.gather(*tasks)
```

### Issue 3: Wasted Worker Pool
- 10 workers available
- Only 1 worker used
- 9 workers sit idle for hours

---

## Recommended Fix

**Option A: CLI submits one task per row**
- CLI reads CSV
- For each row: `enqueue_resolve(candidate_data)`
- 5,200 tasks in queue
- All 10 workers processing in parallel
- Simple, leverages existing `resolve_identity_task`

**Option B: Chunk-based concurrent processing**
- CLI splits CSV into chunks (100 rows each)
- For each chunk: `enqueue_import_chunk(chunk_data)`
- Each chunk task processes rows concurrently with `asyncio.gather`
- Better queue management (52 tasks vs 5,200)
- Still uses all workers

**Option C: Hybrid - Chunk + concurrent rows**
- Like Option B
- Each chunk task limits concurrency with semaphore (e.g., 10 rows at once)
- Prevents memory overload
- Best balance of parallelism and resource management
